# 08 — MVP Boundaries

**Status:** Future planning only  
**Active build priority:** ServicesOS remains priority one.  
**Purpose:** Define what the future SLAI Company Engine should and should not include in its first build.

---

## Main Boundary

The SLAI Company Engine should be planned deeply but built later.

```text
Plan now.
Build later.
ServicesOS remains active priority one.
```

The first version should help Jamie operate and grow SLAI. It should not try to become a complete enterprise command center.

---

## MVP Goal

The MVP should answer:

- What should Jamie focus on today?
- What is blocked?
- What ServicesOS beta issue needs attention?
- What lead/follow-up is worth reviewing?
- What decisions were made?
- What docs or handoffs need updating?

---

## MVP Features

### SLAI OS MVP

- Founder dashboard
- Current priority stack
- Product status cards
- ServicesOS beta status
- Open blockers
- Recent decisions
- Planning doc references
- Manual notes
- Simple next-action recommendations

### GrowthAI MVP

- Manual lead entry
- Lead research notes
- AI-assisted outreach drafts
- Follow-up dates
- Lead status
- Human approval before outreach
- Basic lead fit notes

### Research Agent MVP

- Summarize selected lead
- Identify likely pain points
- Suggest outreach angle
- Store research notes

### ServicesOS Data MVP

- Manually reference ServicesOS beta status
- Later connect lead/quote/customer/job data
- No heavy integration required at first

---

## Not MVP

Do not build early:

- full autonomous outreach
- full CRM replacement
- full project management clone
- full event bus
- large AI agent framework
- automatic customer messaging
- complex dashboard analytics
- investor-style reporting suite
- advanced permissions system
- multi-company command center
- customer-facing GrowthAI dashboard
- deep ServicesOS integration before beta stability

---

## Required Guardrails

Every early feature should follow these rules:

```text
Does this help Jamie focus?
Does this help ServicesOS move forward?
Does this help find or convert customers?
Can this be done manually first?
Can AI suggest instead of acting automatically?
```

If the answer is no, park it for later.

---

## Early Build Shape Later

When promoted, the first build could be a simple internal web dashboard:

```text
Dashboard
Products
Leads
Research
Decisions
Handoffs
Settings
```

It should start with manual entry and light AI assistance before connecting many live systems.

---

## Why Manual First Is Good

Manual-first avoids overbuilding.

It lets Jamie prove:

- which dashboard panels matter
- which lead fields are useful
- which alerts are noise
- which AI summaries save time
- which actions actually help revenue

Then automation can be added where it earns its place.

---

## Promotion Criteria

Do not promote this to active build until:

- ServicesOS beta-critical customer/admin workflows are stable
- wife beta testing has produced useful feedback
- ServicesOS payments are stable enough
- Jamie explicitly says to begin active Company Engine build work
